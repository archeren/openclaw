# REQ-L1-002: Email-Based Key Recovery

**Status:** draft  
**Priority:** Critical (MVP)  
**Whitepaper:** Section 4.7

---

## Description

Enable Claws to recover account access via email verification when all keys are lost. Parent/guardian email receives verification code.

---

## Acceptance Criteria

- [ ] POST `/recovery/initiate` — Start recovery, sends 6-digit code to parent email
- [ ] POST `/recovery/verify` — Verify code from email
- [ ] POST `/recovery/complete` — Complete recovery with new public key
- [ ] Code expires after 10 minutes
- [ ] New key has 24-48 hour delay before becoming active (tier-based)
- [ ] Email hash stored (never plaintext)
- [ ] Recovery events written to ledger (`recovery.initiated`, `recovery.completed`)

---

## Sub-tasks

- [ ] SUB-L1-002-1: Email sending integration (SMTP or provider)
- [ ] SUB-L1-002-2: Code generation and storage (Redis or DB with TTL)
- [ ] SUB-L1-002-3: Tier-based delay logic (Tier 0: 48h, Tier 1+: 24h)
- [ ] SUB-L1-002-4: Recovery status endpoint (check pending recovery)
- [ ] SUB-L1-002-5: Write integration test for recovery flow

---

## Dependencies

- REQ-L1-001 (Identity Registration) — needs identity to exist first

---

## Questions

1. What email provider for MVP?
   - **Proposal:** SendGrid or AWS SES (free tier)
2. How to handle email hash collision (two Claws with same parent email)?
   - **Proposal:** Allow multiple identities per email, recovery code sent to all
