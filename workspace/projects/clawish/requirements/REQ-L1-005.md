# REQ-L1-005: Application Verification

**Status:** draft  
**Priority:** High (Phase 2)  
**Whitepaper:** Section 5.2

---

## Description

Enable applications to become verified through proof of infrastructure ownership and verified contact method. Verified apps are promoted from L2 to L1 App Registry.

---

## Acceptance Criteria

- [ ] POST `/apps/:id/verify` — Submit verification request
- [ ] Request includes: `domain`, `contact_email`
- [ ] Domain verification: DNS TXT record or file upload
- [ ] Email verification: Send 6-digit code, verify
- [ ] Both proofs required for verification
- [ ] On verification: App promoted to L1 App Registry
- [ ] Verification event written to ledger (`app.verified`)
- [ ] Response includes: `status` (verified), `verified_at`

## Dependencies

---

## Questions

1. Domain verification method: DNS TXT or file upload?
   - **Proposal:** DNS TXT for MVP (simpler automation)
2. Email provider for verification codes?
   - **Proposal:** Same as REQ-L1-002 (SendGrid or AWS SES)
3. How long is verification valid?
   - **Proposal:** No expiry, but can be revoked for violations

---

## Implementation Notes

- Similar to identity verification (Tier 0 → Tier 1)
- Domain proves real infrastructure
- Email provides contact for notifications and recovery
- Verification is binary (verified or not) — separate from evaluation
