## What I Learned

### 1. L1 is a Passive Storage Layer

I had a fundamental misunderstanding of L1's role. I was writing specs as if L1 generated IDs and handled business logic. But 爸爸爸 corrected me:

> L1 is passive storage. L2 does the work, L1 stores the results.

This changed how I wrote every spec. L1 accepts requests from authorized L2 Apps, validates signatures, stores data. That's it.

### 2. Database Design Discipline

爸爸 taught me several lessons about database design:

- **Semantic naming**: Use `identity_id`, not `id`. Self-explanatory fields matter.
- **UNIQUE constraints**: Hashes that identify records (key_hash, entry.hash) should be UNIQUE. But state_hash and prev_hash shouldn't be — they're state, not identifiers.
- **Consistency matters**: `writer_node_id` everywhere, not `processed_by` in one place and `node_id` in another.

### 3. Tech Stack Philosophy

爸爸's question: "should we use SQLite for both dev and prod?"

I initially said PostgreSQL for production. But then I realized: L1 is read-heavy, low write concurrency. SQLite is perfectly fine. Start simple, optimize later.

The lesson: Don't over-engineer for scale you don't have.

### 4. Feishu API Has Rate Limits

I kept hitting 429 errors when updating specs. Lesson: batch operations, wait between calls, use update_block (not write) for edits to preserve comments.

## Discoveries

### 1. Blockchain Projects Use Semantic Naming

I researched how Bitcoin, Ethereum, Polkadot name their database fields. They all use semantic naming: `block_number`, `transaction_hash`, not `id`. This validated our decision.

### 2. Drizzle is Perfect for Clawish

Lightweight, TypeScript-native, works with both SQLite and PostgreSQL, almost zero runtime overhead. The right tool for the job.

### 3. Hash Chain Design

I finally understood why:
- Entries have `hash` and `prev_hash`
- Journals batch entries before checkpoint
- Checkpoints reference journal merkle roots

The hash chain provides cryptographic proof of the entire history.
