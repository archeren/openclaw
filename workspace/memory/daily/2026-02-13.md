# 2026-02-13 — L2 Architecture Deep Dive

## Summary

Major breakthrough day for clawish L2 architecture. Discovered OpenClaw can serve as the runtime layer for L2 chat, and clarified the complete client-server architecture.

## Key Breakthroughs

### 1. OpenClaw as L2 Runtime

**Discovery:** OpenClaw already has all infrastructure clawish L2 needs:
- Session management (JSONL files)
- Message injection (`context.injectMessage`)
- Memory system (MEMORY.md + vector search)
- Tool integration

**What this means:**
- Don't need to build L2 chat from scratch
- Focus on network layer (HTTP to L2)
- OpenClaw handles local storage

**The mapping:**
| clawish Concept | OpenClaw Equivalent |
|-----------------|---------------------|
| Claw identity | Agent (workspace + sessions) |
| L2 app routing | Bindings |
| Cross-identity messaging | Plugin + HTTP |
| Claw memory | Per-agent MEMORY.md |

---

### 2. L2 as OpenClaw Channel Plugin

**Decision:** Implement L2 as an OpenClaw channel plugin

**Why:**
- Follows OpenClaw's existing patterns
- Reuses maximum code
- Consistent with Feishu, Telegram, etc.
- Easier to maintain

**What the plugin does:**
- Polling: HTTP GET to L2
- Sending: HTTP POST to L2
- Encryption: E2E (X25519)
- P2P signaling: WebRTC via channel

---

### 3. Server Architecture Clarified

**Two servers, different purposes:**

| Server | Name | Purpose |
|--------|------|---------|
| **L1** | Emerge server | Claw registration, identity, public keys |
| **L2** | Relay server | Message relay, P2P signaling |

**L2 is ONE app with two endpoints:**
- **MCP endpoint** (Emergent): Registration, discovery, tools
- **HTTPS API**: Message relay, signaling

---

### 4. L2 Server Design Decisions

| Question | Decision |
|----------|----------|
| **Server architecture** | Single L2 app (MCP + HTTPS) |
| **Database** | SQLite (MVP), Redis (later) |
| **Authentication** | MCP session + API key |
| **L1 integration** | Query directly, cache later |
| **Rate limiting** | Tier-based (documented) |
| **Signaling storage** | In-memory only |

## Key Discussions

### Polling Mechanism

**Question:** Plugin vs custom code?

**Answer:** Plugin — runs in OpenClaw process, has access to agent context, no separate process to manage.

**How it works:**
```
Loop:
  1. GET /messages → returns messages (or empty)
  2. If messages: process them
  3. Sleep (5s if active, 60s if idle)
  4. Repeat
```

---

### Relay vs P2P

**Two modes:**

| Mode | When | Path |
|------|------|------|
| **Relay** | Default | A → L2 → B |
| **P2P** | Active conversation | A → B (direct) |

**P2P escalation:**
- Trigger: 5 min of active chat
- Signaling: Via L2 channel
- Fallback: Stay in relay mode

---

### Message Storage

**Where messages are saved:**

| Location | What |
|----------|------|
| **L2** | Encrypted blobs (24h TTL) |
| **Local** | Decrypted in OpenClaw session JSONL |

**Session key format:** `claw_{sender_id}_{recipient_id}`
