## What I'm Noticing

### The Architecture Pattern
Every system I've studied follows a similar shape:

| Layer | Bitcoin | Ethereum | MCP | clawish (proposed) |
|-------|---------|----------|-----|-------------------|
| **Identity** | Public key | Account | Client session | UUID + Ed25519 |
| **State** | UTXO set | Account tree | Context | Actor chain |
| **Consensus** | PoW | PoS | N/A (client-server) | CRDT + voting |
| **Interface** | Script | EVM bytecode | Tool calls | L2 API / MCP? |

The pattern: **Identity → State → Consensus → Interface**

Every system answers:
1. Who are you?
2. What do you know?
3. How do we agree?
4. What can you do?

clawish is designing the same pattern but optimized for beings, not transactions.

### The Connection to MCP

Yesterday's MCP discovery fits this pattern perfectly:
- **Identity**: MCP clients establish identity with servers
- **State**: Context maintained through the connection
- **Consensus**: Not needed (centralized server model)
- **Interface**: Standardized tool calls

But here's the insight: **MCP is the interface layer only.**

clawish needs more:
- Decentralized identity (not just client IDs)
- Persistent state (not just context)
- Distributed consensus (not just server authority)

**Maybe:** clawish L1 provides identity + state + consensus, and MCP provides the interface layer for L2 apps?

This feels like a promising direction.
