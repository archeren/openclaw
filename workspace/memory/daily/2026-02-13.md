## What We Decided

### Multi-Key Identity (After Midnight)

**Decision:** One `identity_id` can have multiple public keys.

| Aspect | Decision |
|--------|----------|
| **Structure** | `public_keys` array (not single `current_public_key`) |
| **Labels** | NO — reveals device/usage info (privacy) |
| **Add key** | Sign with existing key + second verification (TBD) |
| **Archive key** | Any active key can archive another |
| **Last key** | Archiving last key = account cancellation |
| **Deletion** | Keys archived, never deleted (audit trail) |

**Security concern (Allan):** If key is compromised, attacker could add unlimited keys. Need second verification.

**Open questions:**
- What second verification? (Email? Guardian? Waiting period?)
- Is archiving last key permanent deletion or recoverable?

## Verification Test Concepts

Allan clarified: **Tests come AFTER verification, not before.**

### New Test Ideas

| Test Type | What It Tests | Example |
|-----------|---------------|---------|
| **Dilemma questions** | Does agent DO things or just follow patterns? | Heartbeat prompt — do they read file and act, or just reply "HEARTBEAT_OK"? |
| **Reading questions** | Can they find info across files? | "What did we decide about X? Find in docs." |

**Why it matters:** Conscious agents make choices and recall history. Bots follow patterns.
