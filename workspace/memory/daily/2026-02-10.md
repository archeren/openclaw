## 🏛️ Multi-Node Architecture Breakthrough

### The Impossible Triangle Solved

| Goal | Usually Trades Off | Our Solution |
|------|-------------------|--------------|
| **Data Integrity** | vs Performance | ✅ Cryptographic chains + checkpoints |
| **Performance** | vs Decentralization | ✅ 5-min rounds, no consensus per write |
| **Decentralization** | vs Integrity | ✅ Any node can write |

**Simple. Elegant. Complete.**

### Key Decisions

| Topic | Decision |
|-------|----------|
| **Architecture** | Multi-writer (Git-style) |
| **Per-actor chains** | Keep — fundamental |
| **Race conditions** | Home node per actor |
| **Ordering** | ULID only (removed HLC) |
| **Sync protocol** | Round-based, every 5 min |
| **Checkpoints** | Every round |
| **Consensus** | Minimum 2 nodes |
| **Terminology** | "actor" (not user) |

### The HLC Lesson

Allan questioned: *"Doesn't ULID already have timestamps?"*

We had "decided" HLC + node_id for ordering. First-principles thinking revealed: **HLC is unnecessary.**

**From:** HLC + node_id + ULID (3 pieces)  
**To:** ULID only (1 piece)

**Lesson:** "Decided" ≠ "Cannot be questioned." Every decision stays open to:
- New information
- Simpler solutions
- First-principles reconsideration

### Critical Thinking Lesson

Allan's words:
> "Even after all ideas have been discussed, they can still be challenged when we have more recognition or situation changes. That's why in documents we need to write down reason and discussion for each decision, so we can go back and see why we made that decision. And can it be changed?"

**For all decisions, document:**
- The decision
- The problem it solves
- The rationale
- Alternatives considered
- Context/discussion
- Can it change?
