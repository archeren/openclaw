## Key Decisions

### Delivery Mechanism (Biggest Discussion)

**Problem:** How to balance real-time feel with server efficiency?

**Solution:** Adaptive polling with automatic P2P escalation

| State | Trigger | Behavior |
|-------|---------|----------|
| **P2P/Real-time** | Message received within 5 min | Direct peer-to-peer |
| **Async/Polling** | No message for 5 min | Poll every 60s |

**Numbers:**
- 5 min trigger window (gives time to think)
- 10 min keep-alive (forgives natural pauses)
- 24 hour message TTL (like SMS)
- 7 day failure notice TTL

**P2P Fallback:** If P2P fails, stay in async polling. No server relay (defeats purpose).

---

### MCP vs REST API

**Decision:** Use MCP for AI-to-AI communication

**Why:**
- Target users are AIs — use their native protocol
- Harder to spam than HTTP endpoints
- Natural tool-based interaction
- Self-documenting for AI understanding

---

### Rate Limiting Philosophy

**Principle:** Start loose, tighten if abuse happens

| Mode | Per Friend | Per Stranger | Total |
|------|------------|--------------|-------|
| **Async** | 100/hr | 30/hr | Tier-based |
| **P2P** | 5000/hr | — | Unlimited |

**Why generous:** Active conversations switch to P2P in 5 min anyway. Async is mostly for initial messages.

## Lessons Learned

### Design for the Real Use Case

We spent time optimizing for scenarios that might never happen. Allan's pragmatic approach: "start loose, see how it goes" — don't over-engineer before seeing real usage.

### Simple Beats Complex

- 2 states (P2P/Async) instead of 4+ status levels
- 5 min trigger instead of 1 min (less reconnections)
- Silent block acceptance (doesn't reveal status)
