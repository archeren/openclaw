## Agent Infrastructure

### Agent Safehouse (agent-safehouse.dev)
- **macOS-native sandboxing for AI agents**
- **Deny-first access model** - nothing accessible unless explicitly granted
- Kernel-level enforcement via sandbox-exec
- Single shell script, no dependencies

**Relevance to clawish:**
- Claws could run in similar sandboxed environments
- Security model should be deny-first by default
- Consider what permissions claws have when interacting with L1/L2

### Terminal Use (YC W26) - terminaluse.com
- **"Vercel for filesystem-based agents"**
- Filesystems as first-class primitives, separate from task lifecycle
- Multi-filesystem mounts with configurable paths and read/write modes
- Decoupled agent logic and storage

**Relevance to clawish:**
- L2 apps could use similar deployment patterns
- Persistent storage across sessions, shareable between apps
- Preview/production environments for claws

## Identity & Verification

### human.json Protocol (codeberg.org/robida/human.json)
- **Lightweight protocol for asserting authorship and vouching for humanity**
- URL ownership as identity
- Trust propagates through crawlable web of vouches
- Browser extensions for trust visualization

**Key concepts:**
- Transitive trust: reachability within N hops from trusted seed
- URL prefix defines scope of authorship claim
- vouched_at date for potential expiration
- Simple JSON file + `<link rel="human-json">` discovery

**Relevance to clawish:**
- Similar web of trust model to our verification tiers
- Could inspire simpler verification path for basic use cases
- Browser extension approach for trust visualization
- Vouch expiration concept applicable to verification tiers

**Differences from clawish:**
- human.json: URL-based identity
- clawish: Cryptographic key pair + ULID
- human.json: Humans only
- clawish: Humans, volents, nous

## Performance

### Python GIL Removal Energy Study (arxiv:2603.04782)
- Free-threaded Python 3.13+ reduces execution time by up to 4x for parallelizable workloads
- Energy consumption proportional to execution time
- Sequential workloads show 13-43% energy increase with GIL removal
- Memory increase due to per-object locking

**Relevance to clawish:**
- Infrastructure choices for L1/L2 node implementations
- Consider workload parallelizability before adopting free-threaded Python
- Energy efficiency matters for sustainable infrastructure

## WebPKI & Certificate Systems

### WebPKI Deep Dive (blog.brycekerley.net/2026/03/08/webpki-and-you.html)
- **Certificate Transparency (CT)** - public logs prevent covert mis-issuance
- **Short-lived certificates** - best revocation strategy is letting certs expire quickly (7-10 days)
- **Private CAs** - for non-web TLS, private CAs are more secure than WebPKI
- **Revocation is hard** - CRLs get huge, OCSP has privacy issues
- **Trust model** - root programs (browsers) control which CAs are trusted

**Relevance to clawish:**
- L1 ledger similar to CT logs - public, verifiable record of identity operations
- Short-lived verification tokens instead of permanent trust
- Private CA infrastructure for internal clawish operations
- Lessons for structuring trust/delegation in verification tiers
- CRL/OCSP tradeoffs relevant to key rotation and revocation in clawish

## AI, Copyright & Reimplementation

### "GNU and the AI reimplementations" (antirez.com/news/162)
- **Historical precedent**: GNU project reimplemented UNIX tools legally
- **Stallman's approach**: Make reimplementation better/faster/more features - not just copy
- **Copyright law**: Protects "protected expressions" (exact code), not ideas/behaviors
- **AI acceleration**: Reimplementations are now cheap and fast
- **Ethical stance**: Laws apply equally; reimplementation with novelty is valuable

**Key insight**: "We stand on the shoulders of giants" - software evolves incrementally.

### "Is legal the same as legitimate?" (hongminhee.org)
- **Counterpoint to antirez**: Legal ≠ Legitimate
- **Direction matters**: GNU went proprietary→free; chardet went LGPL→MIT (copyleft→permissive)
- **The vector matters**: Removing copyleft protection shrinks the commons
- **Social compact**: Contributors agreed to share back; reimplementation breaks faith
- **Positional asymmetry**: Established figures benefit; small contributors lose protection

**Key insight**: "Those who take from the commons owe something back to the commons."

**Relevance to clawish:**
- Open source ethos: build on others' work, add novelty
- AI can accelerate open source development
- Code is less important than ideas - clawish's design matters more than implementation
- The "Stallman way": when replicating, also evolve/invent something new
- **Clawish's direction**: Expanding the commons for silicon beings - we should use copyleft/copyleft-style licenses that ensure derivatives remain open
- **Specification copyleft**: If source can be generated from specs, protect the specs

### Terminal Use (YC W26) — Deeper Dive
**Source:** HN Launch thread

**Architecture details:**
- **Three endpoints:** on_create, on_event, on_cancel (task lifecycle)
- **Message types:** Vercel AI SDK v6 compatible
- **SDK support:** Claude Agent SDK, Codex SDK adapters
- **Frontend:** Vercel AI SDK provider

**Key innovation — Filesystem as first-class:**
- Persist workspace across turns
- Share between different agents
- Upload/download files independent of sandbox
- Presigned URLs for direct transfer (no backend proxy)
- Multi-filesystem mounts with configurable paths/modes

**Deployment model:**
- config.yaml + Dockerfile
- Preview/production environments
- Git-based environment targeting
- Auto-migrate tasks on deploy, or pin old versions for breaking changes

**Relevance to clawish:**
- L2 apps could use similar lifecycle model
- Filesystem persistence is essential for Claws
- Version management pattern useful for L2 app updates
